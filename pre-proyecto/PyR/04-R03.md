## 1. Evaluación general de la propuesta

### 1.1 Validez de la propuesta

La propuesta de `03-R02.md` es **válida** como base de un sistema doc-first para el proyecto. Los 4 documentos propuestos ("Contratos") establecen una separación clara de responsabilidades y siguen una dependencia lógica coherente:

```
B08 → Contrato funcional → Contrato de proceso → Contrato de IA → Contrato técnico → Desarrollo
```

Esta estructura responde adecuadamente a lo identificado en `legado-obsoleto/01-Boceto_B08.md` y `02-R01_r.md`, especialmente:

| Requisito de B08/R01-r | Cubierto por Contrato |
| ---------------------- | --------------------- |
| WA externa como centro operativo | Contrato 1 y 4 |
| Login validado contra WordPress | Contrato 4 |
| Múltiples ejecuciones de IA con roles | Contrato 3 |
| Flujo con aprobación humana previa | Contrato 1 y 2 |
| Gestión de estados y trazabilidad | Contrato 2 |
| Publicación vía API WooCommerce | Contrato 4 |
| Configuración de proveedores IA (CRUD) | Contrato 4 (implícito) |
| Plazo: 18 de mayo de 2026 | Requiere documento adicional |

### 1.2 Suficiencia de la propuesta

La propuesta es **suficiente en estructura pero incompleta en alcance**. Los 4 contratos cubren las áreas principales identificadas en B08, pero existen lagunas críticas que no están contempladas:

| Área crítica | Estado en R02 |
| ------------ | ------------- |
| Objetivos de negocio cuantificables | No cubierto |
| Criterios de aceptación | No cubierto |
| Requisitos no funcionales (rendimiento, disponibilidad) | No cubierto |
| Planificación y hitos | No cubierto |
| Glosario unificado | No cubierto |
| Matriz de trazabilidad | No cubierto |

### 1.3 Adecuación al enfoque doc-first

La propuesta es **adecuada conceptualmente** pero requiere refinamiento operativo:

| Criterio | Valoración | Observación |
| -------- | ---------- | ----------- |
| Documentos como fuente de verdad | ✅ Adecuado | El concepto de "Contrato" es apropiado |
| Dependencia entre documentos | ✅ Adecuado | La secuencia propuesta es lógica |
| Separación de responsabilidades | ✅ Adecuado | Negócio/Proceso/IA/Técnico están bien diferenciados |
| Accionabilidad para IA/desarrollador | ⚠️ Parcial | Falta especificar formato de salida esperado |
| Control de pendientes | ✅ Adecuado | La regla "no inventar" está bien definida |

## 2. Análisis de cada documento propuesto

### 2.1 Contrato 1: Funcional del sistema

| Aspecto | Evaluación |
| ------- | ---------- |
| **Cobertura B08** | Alta. Recoge WA, usuarios, aprobación, límites, relación PDF-producto |
| **Cobertura R01-r** | Media. No incluye explícitamente objetivos de negocio de R01-r (respuesta 1) |
| **Claridad de alcance** | Alta. Los "qué no debe contener" están bien definidos |
| **Pendientes identificados** | Correctos. Funciones admin, criterios aprobación, múltiples productos |

**Problemas detectados:**

1. No incorpora explícitamente los **objetivos de negocio de R01-r respuesta 1**: "reducir tiempo", "eliminar errores", "procesar alto volumen", "información veraz", "fácil de entender", "mejorar posicionamiento en buscadores y LLM".

2. No menciona el **objetivo de SEO/LLM** identificado en R01-r, que es crítico para el contenido generado.

3. La relación con el CRUD de proveedores IA (R01-r respuesta 8) no está clara: ¿es funcional o técnico?

**Recomendación:** Ampliar la sección "Debe tratar" para incluir objetivos de negocio explícitos y criterios de calidad del contenido (SEO, legibilidad).

---

### 2.2 Contrato 2: Proceso, estados, errores y trazabilidad

| Aspecto | Evaluación |
| ------- | ---------- |
| **Cobertura B08** | Alta. Cubre flujo, cancelación, rechazo, aprobación, fallos, log |
| **Cobertura R01-r** | Media-Alta. Podría integrar mejor el requisito de log con identidad de usuario |
| **Claridad de estados** | Alta. La distinción de momentos está bien planteada |
| **Pendientes identificados** | Correctos. Nombres de estados, formato de log, referencia de error |

**Problemas detectados:**

1. No especifica **qué constituye un "paso completado"** vs "paso iniciado" de forma medible.

2. La **referencia de error al usuario** (R01-r respuesta 14, B08 sección 36) necesita formato concreto: ¿código? ¿ID de log? ¿ambos?

3. No menciona **tiempos de espera o timeout** por paso, relevante para el botón de cancelación.

4. El **log histórico** (B08 sección 39) necesita ubicación y formato definidos: ¿tabla WordPress? ¿archivo? ¿ambos?

**Recomendación:** Añadir sección específica para "Métricas de proceso" que incluya timeouts, formato de referencia de error y ubicación del log.

---

### 2.3 Contrato 3: IA, extracción y contenido de producto

| Aspecto | Evaluación |
| ------- | ---------- |
| **Cobertura B08** | Media-Alta. Cubre roles, extracción, contenido, formulario |
| **Cobertura R01-r** | Media. No integra completamente el CRUD de proveedores (respuesta 8) |
| **Claridad de entradas/salidas** | Alta. La estructura de "debe tratar" es clara |
| **Pendientes identificados** | Correctos pero numerosos. Muchos bloqueantes |

**Problemas detectados:**

1. **Pendientes bloqueantes críticos:** Los campos WooCommerce (R01-r respuesta 11) están "pendientes de definición detallada". Esto es **bloqueante** para el Contrato 3 y, por extensión, para todo el desarrollo.

2. **Proveedor de IA:** R01-r respuesta 8 indica que debe haber CRUD de proveedores, pero el Contrato 3 no especifica cómo afecta esto a los prompts (¿formato compatible con múltiples proveedores?).

3. **Calidad de extracción:** B08 sección 37 identifica "calidad del PDF" como dependencia crítica, pero no hay criterios de aceptación para extracción incompleta/incorrecta.

4. **SEO/LLM:** R01-r respuesta 1 menciona "redacciones orientadas a mejorar posicionamiento", pero no hay requisito concreto en el Contrato 3.

**Recomendación:** Este contrato tiene la mayor densidad de pendientes bloqueantes. Requiere priorización explícita: qué debe resolverse ANTES de desarrollar vs qué puede resolverse DURANTE el desarrollo.

---

### 2.4 Contrato 4: Técnico de arquitectura e integraciones

| Aspecto | Evaluación |
| ------- | ---------- |
| **Cobertura B08** | Alta. Cubre WA, WordPress, WooCommerce, autenticación, almacenamiento |
| **Cobertura R01-r** | Media-Alta. Integra stack PHP-React, mismo servidor, tabla WP a crear |
| **Claridad técnica** | Alta. Los "debe tratar" son implementables |
| **Pendientes identificados** | Correctos. Endpoints, nonces, organización de subcarpeta |

**Problemas detectados:**

1. **Framework no decidido:** R01-r respuesta 4 menciona `analizador/_otro-proposito/PHP-React-Framework/` como opción, pero "alternativa a determinar". El Contrato 4 debería requerir esta decisión antes de desarrollar.

2. **Tabla WordPress:** R01-r respuesta 12 dice "debe crearse", pero no especifica estructura mínima. El Contrato 4 debería incluir schema mínimo.

3. **CRUD de proveedores IA:** R01-r respuesta 8 requiere CRUD para proveedores, APIs y límites. No está claro si esto es parte del Contrato 4 o del 1.

4. **Nonces/CSRF:** Mencionado como pendiente, pero es crítico para seguridad. Debería ser decisión temprana.

**Recomendación:** Mover decisiones de seguridad (nonces, CSRF, autenticación) a "no negociables" en lugar de pendientes.

## 7. Conclusión final

### 7.1 Validez del enfoque doc-first

**La propuesta de `03-R02.md` es válida** como base de un sistema doc-first para el proyecto. La estructura de 4 Contratos con dependencia secuencial es apropiada y responde a las necesidades identificadas en B08 y R01-r.

### 7.2 Condición de suficiencia

**La propuesta NO es suficiente en su estado actual.** Requiere:

1. **Incorporación de objetivos de negocio** de R01-r respuesta 1 (especialmente SEO/LLM).
2. **Resolución de pendientes bloqueantes** antes de desarrollar (campos WooCommerce, framework, schema de tabla).
3. **Documentación adicional mínima**: criterios de aceptación, glosario, planificación.

### 7.3 Recomendación de implementación

| Fase | Acción | Salida esperada |
| ---- | ------ | --------------- |
| **Fase 0** | Resolver pendientes bloqueantes | Campos WC, framework, schema definidos |
